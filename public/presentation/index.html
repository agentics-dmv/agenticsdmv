Then WhatsApp Closed the Door.

How a health monitor turned 20 gateway restarts into a rate-limited phone number.

Three hours of debugging. SSM port forwarding drops WebSockets. Every drop reset the QR flow. A Cloudflare tunnel required editing openclaw.json — wrong key, service crashes. The health monitor turned every restart into a WhatsApp reconnect, a 401, exponential retries. Twenty restarts. Hundreds of failed auth attempts. Phone: "Can't link new devices at this time." [If asked about the SSM-WebSocket flakiness fix] The OpenClaw docs document Tailscale Serve as the stable remote-access path: `tailscale serve --bg http://localhost:18789` puts the Control UI behind a stable HTTPS URL on your tailnet, no port forward, no WebSocket re-handshakes, identity headers passed through. That's the right answer if SSM port-forward keeps dropping you. We didn't swap to it because Telegram replaced the need — but it's the documented escape hatch.

This is the core insight. Six markdown files on disk, concatenated into the system prompt on every message. To change the personality — edit SOUL.md, run make ship, send a message. The next response follows the new instructions. You're writing software. It just happens to be in English.

SOUL.md

Sharp. Lowercase. Slightly unimpressed. Efficient.

This is the actual personality spec. The model talks like a Gen Z teenager who finds your question mildly beneath them but answers correctly anyway. Not a profound technical achievement. But the bot felt like someone with opinions, not a customer service form.

Memory Has Layers.

Raw sources stay locked. The wiki is the working layer. Agent Governance — the workspace files — runs on every turn.

The temptation is to load everything into context. At 500 research files, that's over a million tokens — the entire context window, nothing left for conversation. The spec named this failure mode the "Token Crusher." Instead: a flat JSON index, ~15k tokens, fetched on demand. Load the card catalog, not the library. The model finds the entry, fetches the document, reads it.

The Idea.

A voice memo becomes a wiki node. Nothing about the implementation was this clean.

The concept is simple. The execution had seven steps, each one that could fail independently. Let me show you the pipeline.

Nine Steps.

Transcription error in step 3. Didn't surface until step 6.

Upload to S3. Start a Transcribe job. Poll until done. Download transcript. Create source page. Run LLM integration pass. Commit everything. Index rebuilds. Notify. Nine steps. The error lives in step 3 — but you don't find out until step 6 when the wiki page says your son was born in Lagos, Nigeria. More on that in Act 3.

The Pipeline, End to End.

Telegram → S3 → Transcribe → markdown → GitHub → knowledge base. $0.024/min. 60–120 seconds. Bot acknowledges in 2.

The acknowledgment is non-optional. "Got it, transcribing..." costs nothing and prevents two minutes of silence reading as failure. The pipeline runs in the background. The bot handles other messages while it waits. [Mechanic, if asked] No Lambda. The pipeline runs in-process on the EC2 instance: Telegram webhook hands the audio to a tool inside the gateway, which uploads to S3, calls `aws transcribe start-transcription-job`, polls for completion, fetches the result, commits the markdown to the openclaw-research repo, pushes. GitHub Actions rebuilds the index. SSM RunCommand notifies the bot. Lambda would have been cleaner but added more pieces than it removed for a single-user system.

The Keys That Weren't There.

Three paths for API keys. Only one reached the running gateway process.

First research run. Script called the API. Nothing came back. The .env file: PARALLEL_API_KEY= — empty string. The key had been in SSM Parameter Store since day one. Nothing had ever pulled it into the environment. Then the real gap: .env was correct, .profile source line was there, interactive sessions picked up the keys fine — but the gateway runs as a systemd user service. Systemd doesn't source .profile. One line in the service file: EnvironmentFile=/home/ubuntu/.openclaw/.env. Verified from /proc//environ. This is in the takeaways.

Same Instructions. Different Model Class.

Nova 2 Lite ignored the workspace files. Sonnet 4.6 followed them. No prompt engineering closes that gap.

The log: "agent model: amazon-bedrock/global.amazon.nova-2-lite-v1:0." Nova 2 Lite. Not Sonnet 4.6. The model upgrade had never been saved into the running config. Every session since day one had been the wrong model. Nova 2 Lite is optimized for speed — AWS's own docs say "system prompt adherence can decline slightly as context increases." With 6,650 tokens of workspace files on every turn, "slightly" is not the word. This is not a tier difference. It's a model-class difference. No amount of prompting closes it. [Mechanic, if asked] The model ID matters. The right value is `us.anthropic.claude-sonnet-4-6` — the `us.` prefix is the cross-region inference profile, which routes the call across us-east-1, us-east-2, and us-west-2 to dodge per-region throttling. The raw `anthropic.claude-sonnet-4-6` works but stalls under load. New AWS accounts also have to opt in to Anthropic models in the Bedrock console first; that's a separate gate from the IAM role.

One sed Command.

sed -i "s|nova-2-lite-v1:0|us.anthropic.claude-sonnet-4-6|" openclaw.json

[gateway] ready (9 plugins, 1.9s)

One string replacement. Gateway restart. The personality took hold. The protocols started working. The capabilities unlocked. The lesson: making instructions louder doesn't help. Checking the logs does.

Act 3

What Held

The right model was running. The pipeline was deployed. The first real test produced wrong data, and the bot was about to fix its own code without telling anyone. What held wasn't the absence of breakage. It was the structure that caught it.

Correct Reasoning. Bad Data.

wiki/entity/rio.md
Birthplace: Lagos, Nigeria

speculative: was Ford living in Nigeria at the time of the birth?

AWS Transcribe heard "Lagos" where Richmond was said.
The LLM had no way to know. It reasoned correctly from bad data.
"Thadeus" became "Thaddeus" everywhere. Profile created as will-prior.md.

The pipeline worked. Within three minutes, five wiki pages were created, committed to GitHub, bot replied with a link. One page said my son was born in Lagos, Nigeria. Reasonable inference from bad transcription. This isn't a bug to fix with prompting. Speech-to-text has error rates. LLM passes amplify them. The mitigation is a workflow: bot creates drafts, human reviews, corrections commit with provenance. Source transcripts stay immutable. Derived pages evolve.

One Parameter. Deleted an Entire Stage.

"auto" → "text" deleted the Bedrock reformatting stage and its silent failure mode.

First deep research results rendered as 250 lines of raw JSON on GitHub. The script was submitting with output_schema: { type: "auto" } — structured JSON, right for batch pipelines, wrong for readable reports. There was a second stage that called Bedrock to reformat it. When that failed silently, the fallback wrote raw JSON directly to the file. The fix: Parallel's API has a "text" mode. Returns markdown with inline citations, no schema needed, no downstream formatting step. One word change. 36 lines of bash deleted.

What It Is Now.

This is the picture you'd draw on a whiteboard. The next slides are the part you wouldn't have thought to ask about.

One EC2 instance. One Node.js gateway. Port 18789, localhost only. All external access through SSM. On every message: six markdown files assembled into a prompt, session history appended from JSONL on EBS, Bedrock called via IAM role, response returned. The instance has 4GB of RAM and runs no inference. It's a prompt assembler and a caller. The model is in Bedrock. [Mechanic, if asked about cost] Most of that prompt is the same six files turn after turn. Anthropic prompt caching splits the prompt at a stable boundary — workspace files cached, conversation tail volatile. Cache writes are 1.25× input, cache reads 0.1×. With a 5-minute default TTL on Anthropic-via-Bedrock, bursty Telegram traffic pays the write penalty repeatedly. The OpenClaw config knob is `agents.defaults.params.cacheRetention: "long"` (1-hour TTL on direct Anthropic; Bedrock pass-through behavior is documented as ambiguous). Combined with `contextInjection: "continuation-skip"` and a 55-minute heartbeat to keep the cache warm, that's a 10×+ Bedrock cost lever with no prompt changes. Bug 14 in the writeup goes into the numbers — 92% of our 30-day spend was cache writes.

The Cache-Write Storm.

$184.71 of $200.68 was cache writes. The bill isn't the model — it's the cache.

Thirty days of Bedrock. Two hundred dollars list price. Ninety-two percent of it — cache writes. Output tokens were three dollars. The model isn't the bill. The cache is the bill. Cost Explorer shows seventy-three after credits, but credits aren't strategy. The lever is on this slide: cacheRetention long, continuationSkip on context injection, heartbeat every fifty-five minutes to keep the cache warm. The five-minute default TTL on Anthropic-via-Bedrock loses the cache between bursty messages, so almost every turn pays the write penalty and almost none reaps the read benefit. Three config lines, ten-times Bedrock cost lever, no prompt engineering. The number on the right — ten dollars per month — is projected, not measured. We haven't shipped the change yet. That's the next experiment.

Why The Cache Expires First.

Same six turns. Same conversation rhythm. Different TTL. Roughly 4× cost difference.

Telegram conversations are bursty. A message every ten minutes is normal. The default 5-minute cache TTL means the cache expires before the next message arrives — every turn pays the write cost, none of them get the read discount. Top timeline. Six writes, sixty-nine cents. Switch cacheRetention to long and the TTL goes to one hour. Now you write once, read five times, refresh at fifty-five minutes via heartbeat. Bottom timeline. Eighteen cents — actually seventeen and a half, the slide rounds. Same conversation. Four times cheaper. One caveat in the footer: OpenClaw's docs say the one-hour upgrade is documented for direct Anthropic hosts, and they're silent on Bedrock pass-through behavior. So before claiming the win, check CloudWatch's CacheWriteInputTokenCount and CacheReadInputTokenCount metrics per model ID. If the writes still dominate after the change, Bedrock isn't honoring the longer TTL and we're back to negotiation.

Three Knobs That Move the Bill.

{
  "agents": {
    "defaults": {
      "params": { "cacheRetention": "long" },
      "contextInjection": "continuation-skip",
      "heartbeat": { "every": "55m" }
    }
  }
}
cacheRetentionTells the provider to hold the prefix cache for an hour instead of five minutes. The TTL lever.
contextInjectionSkips re-injecting the workspace bootstrap on continuation turns. Stops the prefix from being re-emitted unnecessarily.
heartbeat.everyBackground ping just under the cache TTL window. Keeps the cache warm across idle gaps so the next user message hits a read, not a write.

Three lines. Roughly 10× Bedrock cost lever. No prompt engineering.

Three lines in openclaw.json. cacheRetention long — extend the TTL from five minutes to one hour. contextInjection continuation-skip — don't re-emit the workspace bootstrap when a turn is a continuation. heartbeat every fifty-five minutes — a background ping just under the cache window so the cache stays warm even when you're not talking to the bot. Together, that's a roughly ten-times cost lever on the part of the bill that actually scales. No prompt rewrites. No model swap. The architecture stays exactly the same; the cost shape changes. This is the slide to take a picture of if you're going to try this. The Bedrock pass-through caveat from the previous slide still applies — measure before claiming the win.

Six Things to Steal.

  1. Work with the platform's design, not against it. CLI pairing works. WebSocket tunnels don't.
  2. Check what model is actually running before touching the instructions.
  3. Systemd doesn't source .profile. If your gateway has no env vars, check the unit file.
  4. Simpler pipelines fail louder. "auto" to "text" deleted a stage and its silent failure mode.
  5. Your agent needs an onboarding doc. Write it in plain English. It will follow it.
  6. ship = push + deploy. Close the gap or you'll forget it exists.
Each one is a real failure that happened and got fixed. Not theoretical. The goal is that you leave with at least one of these applying to something you're building.
