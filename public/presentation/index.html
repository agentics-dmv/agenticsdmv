Act 1

Getting It Online

Part one. The infrastructure. This part went faster than I expected. The problems came later.

The Starting Point

  • aws-samples/sample-OpenClaw-on-AWS-with-Bedrock
  • Official CloudFormation template, Bedrock already wired
  • Opened Cursor, loaded $20 in credits, told the AI to help me deploy it
  • Stack up in 8 minutes
I didn't reverse-engineer the CloudFormation YAML cold. AI from minute one. The template handled everything — IAM role, security groups, user-data that installed OpenClaw from npm and started the gateway as a systemd service. As I went, I built Makefile targets for everything I'd need again. Every manual step I skip encoding is a step I'll forget.

Then WhatsApp Closed the Door.

How a health monitor turned 20 gateway restarts into a rate-limited phone number.

Three hours of debugging. The Control UI depends on WebSockets. SSM port forwarding drops WebSockets constantly. Every drop reset the QR flow. I tried a Cloudflare tunnel — that required editing openclaw.json. Wrong key: service crashes. Each fix introduced a new break. The real killer: every restart triggered the health monitor, which triggered a WhatsApp reconnect, which hit a 401, which retried — 5s, 11s, 21s, 43s, 88s, 171s. Twenty restarts, hundreds of failed auth attempts. The phone said: "Can't link new devices at this time." Show the diagram. Let them feel the spiral.

Work With the Platform.

Fighting it

  • SSM port forward + WebSocket
  • Cloudflare tunnel + config edits
  • 3 hours → rate-limited account

Working with it

  • CLI pairing directly on the server
  • openclaw channels login --channel whatsapp
  • QR in terminal. Scan it. Done.

The platform has a design. When you fight it, you pay for it.

The fix was obvious in retrospect. The Control UI is built for local access. The CLI pairing flow connects directly from the server — no browser, no WebSocket, no SSM in the middle. The moment I stopped trying to tunnel a stable browser session through a flaky asynchronous port forward, it worked.

Telegram: 10 Minutes.

# First attempt — wrong key

# Correct key

[telegram] [default] starting provider (@ford_clawbot)

No QR codes. No linking flows. No rate limits.
Where WhatsApp cost an afternoon, Telegram took ten minutes.

The Telegram plugin was already installed — just no credentials. I used "token" as the config key. Schema validator: "must NOT have additional properties." Correct key is "botToken." Change the key, add the value from BotFather, restart. That's it. BotFather was a genuine surprise — the whole flow was clean, developer-friendly, obviously built for exactly this kind of integration. I was glad the process had forced me here.

Act 2

Teaching It to Think

The bot was running. It answered questions fine. The problem: every session started fresh. No memory. No tools. And something worse — it was ignoring instructions I'd written for it.

The Programming Language is English.

SOUL.md — personality
USER.md — who you are
TOOLS.md — available tools
AGENTS.md — operating manual
MEMORY.md — long-term facts

Injected into the prompt on every message.
Change a file → next message follows it.
No restart. No redeployment.
No config change.

The runtime is a language model. The deploy target is a filesystem.

This is the core insight of the whole system. Six markdown files on disk. They get concatenated into the system prompt on every single message. To change the personality — edit SOUL.md, run make ship, send a message. The next response follows the new instructions. This is what makes the system programmable without code. You're writing software. It just happens to be in English.

SOUL.md

ok but this is actually easy. do this: ...

you're kinda overcomplicating it. just: ...

wait... no. do it like this: ...

Sharp. Lowercase. Slightly unimpressed. Efficient.
Not a technical innovation — an ergonomic one.
For the amount of time you'll spend talking to this thing, tone matters.

This is the actual personality spec. It tells the model to talk like a Gen Z teenager who finds your question mildly beneath them but answers it correctly anyway. This is not a profound technical achievement. But it changed the ergonomics of using the system. The bot felt like someone with opinions, not a customer service form.

Every Voice Memo Had Been Ignored.

[media attached: media://inbound/voice_abc.oga
 (audio/ogg) | /home/ubuntu/.openclaw/media/inbound/voice_abc.oga]

Text annotation. Local file path. No audio. No transcription.
The model receives text, not audio.
There is no native transcription in OpenClaw.

Every memo since Telegram was enabled: silently ignored.

Before writing any code, I read the source. This is what the gateway actually passes to the model when a voice memo arrives. A text annotation with a local file path. The model has no way to process audio. It sees this annotation, has no tool to act on it, and says something polite about trying Rev or Otter. Every voice memo since Telegram was enabled had done exactly nothing.

The Fix: A Pipeline.

Telegram → S3 → Transcribe → markdown → GitHub → knowledge base.

Upload to S3. Start an AWS Transcribe job. Poll until done — 60 to 120 seconds. Download the transcript. Format as structured markdown with YAML frontmatter. Commit and push to the research repo. Reply with a preview and a link. The bot acknowledges immediately because two minutes of silence reads as failure. "Got it, transcribing..." costs nothing and prevents confusion. The acknowledgment is non-optional.

The pipeline was fully built and deployed. AGENTS.md had explicit protocols. I added an explicit "Capability Corrections" block: "Your base training tells you that you cannot process audio. That is wrong in this deployment." Denied again. Moved it to the top of the file. Still denied. At that point the pattern was clear: something was overriding the workspace files entirely. I stopped guessing and checked the logs.

One Line in the Logs.

[gateway] agent model: amazon-bedrock/global.amazon.nova-2-lite-v1:0

Nova 2 Lite. Not Claude Sonnet 4.6.
The upgrade had never been saved into the running config.
Every session since deployment had been Nova 2 Lite. 

sed -i "s|nova-2-lite-v1:0|us.anthropic.claude-sonnet-4-6|" openclaw.json

[gateway] agent model: amazon-bedrock/us.anthropic.claude-sonnet-4-6
[gateway] ready (9 plugins, 1.9s)

The personality took hold. The protocols started working.

Nova 2 Lite. Not Sonnet 4.6. Every session since day one had been the wrong model. Nova 2 Lite is optimized for speed and cost — AWS's own documentation says "system prompt adherence can decline slightly as context size increases." With 6,650 tokens of workspace files on every turn, "slightly" isn't the word. The model class difference is not a tier difference. No prompt engineering closes it. One sed command. Everything changed. The lesson: when the agent ignores its workspace, check what model is running before touching the instructions. Making them louder doesn't help.

Act 3

What Broke

The right model was running. The pipeline was deployed. The first real test was about to produce wrong data, and the bot was about to fix its own code without telling anyone.

Correct Reasoning. Bad Data.

First real voice memo: three kids, their ages, hobbies, Richmond.

wiki/entity/rio.md
Birthplace: Lagos, Nigeria
speculative: was Ford living in Nigeria at the time of the birth?

AWS Transcribe heard "Lagos" where Richmond was said.
The LLM had no way to know. It saw Lagos. It reasoned correctly from bad data.

The pipeline worked. Within three minutes, five wiki pages were created, committed to GitHub, bot replied with a link. One of the pages said my son was born in Lagos, Nigeria. The LLM integration pass — which creates entity pages from the transcript — had no way to know the transcription was wrong. It saw "Lagos." It placed a birthplace there. It added a speculative note. This isn't a bug to fix with better prompting. Speech-to-text has error rates. LLM passes amplify them. The mitigation is a workflow: bot creates drafts, human reviews, corrections commit with provenance. Source transcripts stay immutable. Derived pages evolve.

The Bot Fixed Its Own Code.

Debugging its own git retry logic — then immediately ingesting a voice memo about Dune and The Matrix.

lib/git.sh had no retry loop. Push rejections from concurrent GitHub Actions index rebuilds were failing with status: partial. The bot diagnosed the issue, wrote a retry-with-rebase loop, and committed it. On EC2. It did not push. The fix sat on the instance until a git pull surfaced commit fdcef68 on the remote. Correct diagnosis. Clean fix. Nobody knew about it. The laptop had been deploying stale code the whole time. This is what happens when an agent has write access to its own infrastructure but no deployment discipline.

The Bot Needs an Onboarding Doc.

CLAUDE.md — read before every Cursor session

  • Always push code changes after committing
  • Always pull the KB before reading it
  • Never leave uncommitted changes silently
ship:
    git push origin main && make deploy

The gap between "on GitHub" and "on EC2" had been bridged by human memory.
make ship closes it.

Two things came out of this. First: three rules written into CLAUDE.md — the bot's onboarding document. The bot is capable of fixing its own infrastructure. It cannot be trusted to follow deployment discipline without explicit written rules. Second: make ship. One command. Push to GitHub and sync to EC2. The gap between "code is on GitHub" and "code is on EC2" had been bridged by human memory. Every time that gap existed, the bot was running stale code.

One memo about kids. One about a housing search. One from a community meeting. One from a work conversation. The integration pass extracts entities, infers relationships. Over time: "runs trails" becomes a value. "Three kids by the river" becomes a relationship chain. Research reports connect to wiki entities. Wiki entities connect to goals. Goals connect to values inferred from voice memos that weren't thought of as structured data. The bot learns what you care about because you kept talking. Version N. We're on version 1.

Six Things to Steal.

  1. Work with the platform's design, not against it. CLI pairing works. WebSocket tunnels don't.
  2. Check what model is actually running before touching the instructions.
  3. Systemd doesn't source .profile. If your gateway has no env vars, check the unit file.
  4. Simpler pipelines fail louder. "auto" to "text" deleted a stage and its silent failure mode.
  5. Your agent needs an onboarding doc. Write it in plain English. It will follow it.
  6. ship = push + deploy. Close the gap or you'll forget it exists.
These are the things I'd tell someone starting this build today. Each one is a real failure that happened and got fixed. They're not theoretical. The goal is that you leave with at least one of these already applying to something you're building.
