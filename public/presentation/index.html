Building a 24/7 AI Assistant on AWS

What worked. What didn't. Why the failures teach more.

AWS Meetup — Ford Prior

Welcome everyone. This talk is about what actually happens when you take something that looks easy on a blog post — run a personal AI assistant on AWS — and go build it for real. Spoiler: it's not hard, but it's not what you think either. I want to walk you through the actual experience over three days of building, including everything that broke and why.

The Stack

t4g.medium · Graviton · 30GB EBS · no public ports · Claude Sonnet 4.6 via Bedrock

The stack is simple by design. One EC2 instance — ARM64, cheap, quiet — runs a Node.js gateway that holds a connection to Telegram and calls Bedrock on every message. No database. No load balancer. No NAT gateway. The CloudFormation template asked a few questions and built the whole thing: instance role, security groups, a user data script that pulled OpenClaw from npm and started the gateway as a systemd service. The destination. Getting here took longer than the diagram suggests.

The WhatsApp Death Spiral

How a health monitor turned 20 gateway restarts into an account ban.

Here is what actually happened. Problem 1: SSM port forwarding drops WebSocket connections constantly, and the OpenClaw Control UI depends on WebSockets for everything — QR generation, status updates, the pairing flow. Every dropped connection was a reset. Problem 2: I tried a Cloudflare tunnel — that required touching openclaw.json. Touching openclaw.json is where things went wrong. The service crashed. Problem 3: Without SSH, the only way to edit the config was SSM RunCommand — Python one-liners, JSON escaped into shell commands, 10-second round trip, no interactive feedback. Problem 4: Every time the gateway restarted, the health monitor saw WhatsApp in a stopped state and triggered a reconnect, which hit a 401, which retried — after 5 seconds, 11, 21, 43, 88, 171. With more than twenty restarts in one afternoon, that was hundreds of failed auth attempts. WhatsApp noticed. The phone said: "Can't link new devices at this time."

The Lesson

"OpenClaw's Control UI is built for local access.
The gateway listens on localhost.
The QR flow expects a stable browser on the same machine."

Fighting that architecture → WebSocket drops, config crashes, rate limit
Working with it → CLI pairing (openclaw channels login) worked immediately

The platform has a design. Work with it.

The lesson — not a surprise in retrospect — is that the Control UI is built for local access. SSM port forwarding is asynchronous and drops connections, which makes it exactly wrong for anything that needs a persistent WebSocket. Fighting that architecture produced every failure. The CLI pairing flow — run directly on the server, connecting to WhatsApp's servers with no intermediary — worked immediately when I finally tried it. The platform has a design. Work with it.

The bot was running. It answered questions, held a conversation, and if you asked it something simple — the weather in Richmond, how to fix a bash script — it was fine. It remembered nothing. The first fix was MEMORY.md — a hand-curated markdown file of long-term facts. About 1,000 tokens. Enough for basics. The second problem was deeper. What do you do when you've run deep research on a topic — thirty minutes of web synthesis — and want to find it six weeks from now? MEMORY.md can't hold 500 research reports. The context window can, technically — Claude Sonnet 4.6 has a 200,000-token window — but loading the full corpus on every message is how you make the assistant slow, expensive, and eventually incoherent as the context fills and the gateway starts compacting conversation history to make room.

The Index, Not a Vector DB

Raw sources stay locked. The wiki is the working layer. Agent Governance — the workspace files — runs on every turn.

The answer was an index. Not a vector database. Not embeddings. A flat JSON file. At 500 research documents, the index is about 15,000 tokens. Load it on demand, let the model reason over it, return the matching reports. The architecture spec for this — written before any code — named the alternative the "Token Crusher": the temptation to auto-load everything. The architecture explicitly prevents it. Research files are cold storage. The index is the card catalog. The agent finds the entry, fetches the document, reads it.

Querying the Knowledge Base

The index returns the matching entry. The agent fetches the document. The answer is grounded in the source.

Here's what the architecture looks like from the phone. You ask a question, the agent checks the index, finds the matching research entry, pulls the full document, and answers from it. The response is grounded — not synthesized from training data, but read from something you published. This is what the flat JSON index enables: cheap lookup at scale, deep reads on demand, no embeddings infrastructure to maintain.

GitHub Actions Wiring

  • OIDC provider → GitHub Actions assumes an AWS role
  • Trust policy scoped to pushes from the research repo's main branch
  • Workflow: markdown push → rebuild index → Telegram notification via SSM
  • Full loop in 2–10 minutes. Bot handles other messages while it waits.

One session. Never opened the AWS console.

Wiring the GitHub side of this took one session without opening the AWS console: an OIDC provider for GitHub Actions to assume an AWS role, a trust policy scoped tightly to pushes from the research repo's main branch, a workflow that rebuilds the index on every markdown push and sends a Telegram notification back to the instance via SSM. The whole loop — research published, index rebuilt, notification delivered — runs in two to ten minutes. The bot handles other messages while it waits.

Research Arrives in Telegram

Index rebuilt. Notification delivered. The loop closes in under ten minutes.

This is the feedback loop in action. A research task runs, the results push to the GitHub repo, Actions rebuilds the index, an SSM command sends a notification to the Telegram bot, and the bot delivers it to the user. Under ten minutes from trigger to delivery. The whole system confirms itself — no polling, no manual refresh, no "did it work?" It tells you.

The Keys That Weren't There

Three paths for API keys. Only one reached the running gateway process.

Building the research pipeline exposed a quiet failure that had been there since the beginning. The first test ran. The script called the research API. Nothing came back. The .env file on the instance had one line for the research key: PARALLEL_API_KEY= — empty. The key had been stored in AWS SSM Parameter Store months earlier. Nothing had ever pulled it into the instance's environment. The gateway had been running without it the entire time, silently failing any research call. Then came the real gap: the .env file was correct, the .profile source line was in place, interactive sessions would pick up the keys — but the gateway runs as a systemd user service, and systemd doesn't source .profile. It uses its own flat environment. One line added: EnvironmentFile=/home/ubuntu/.openclaw/.env. Daemon reload. Restart. Verified directly from /proc//environ. The env file and the service had been disconnected from each other since day one.

What OpenClaw Does With Voice Memos

The idea. A voice memo becomes a wiki node. Nothing about the implementation was this clean.

Voice memos are how you capture things without stopping — walking to a meeting, in the car, picking up kids. The voice memo pipeline started with a question: what does OpenClaw actually do when a voice memo arrives? The source code gave an answer nobody expected. When Telegram sends a voice file, the gateway downloads it and appends an annotation to the prompt: [media attached: media://inbound/voice_abc.oga]. That's it. The LLM receives a text annotation with a local file path. It does not receive the audio. There is no native transcription. Every voice memo since Telegram was enabled had been silently ignored. The audio sat on disk. The annotation sat in the prompt. The model said something polite and moved on.

The Pipeline, End to End

Telegram voice message → S3 → Transcribe → markdown → GitHub → knowledge base.

Here's the full pipeline. A voice message arrives in Telegram, the gateway downloads it to the EC2 instance, the script uploads it to S3, starts a Transcribe job, polls until complete, downloads the transcript, formats it as structured markdown with YAML frontmatter, and pushes it to the research repo where GitHub Actions picks it up and rebuilds the index. Seven steps. Each one can fail independently.

Skills as Operational Memory

Same tokens, deeper thinking. Without skills, the AI explores broadly and fails often. With skills, the early branches are pre-solved.

Throughout this whole process, I kept stopping to do something that felt tangential but turned out to be one of the most important patterns in the build. Every time I figured out how to do something, I asked the AI to write a skill for it. A skill is a reusable instruction file — markdown, stored locally, loaded into Cursor's context on demand. After I learned how to deploy through SSM, I had the AI write an openclaw-deploy skill that captured every step. Next week, when I need to do something I've done before, the skill file means the AI doesn't start from scratch. It picks up where the last session left off. No rediscovery. The system compounds instead of resetting.

The Bot That Fixes Itself

11:11 PM: voice memo in, transcribed. 11:12 PM: "running deep research now." 11:15 PM: report published, link delivered, brief on what it covers.

One thing that struck me immediately, once Sonnet 4.6 was running and the personality was active: the bot was unusually ready to fix things itself. When something broke, the bot didn't just report the error and wait. It would reason through the failure, propose a fix, and often apply it — patching a script, adjusting a config value, retrying a command sequence. It would keep working a thread. Absorb errors. Having the token wasn't enough — the bot would fix something, commit it locally, and not push. The Machine That Learns

Part three: the pipeline worked. The data was wrong. The bot fixed its own code and didn't push it. Three days of catching errors and feeding them back. Then the graph started growing.

Pipeline Worked. Data Was Wrong.

Nine steps. The transcription error lived in step 3 and didn't surface until step 6.

The pipeline worked. Ford dictated a voice memo about his three kids — their ages, their hobbies, the youngest one's obsession with unicorns and scooters — and within three minutes the bot had created five structured wiki pages, committed them to GitHub, and replied with a link. The name was wrong too. The transcript rendered "Thadeus" as "Thaddeus" — double-d — and every downstream page propagated the misspelling. Research Output: Raw JSON

One parameter change deleted an entire pipeline stage and its silent failure mode.

Separately, the first deep research results had been rendering as raw JSON on GitHub. 250 lines of nested objects, arrays, and escaped strings. Not a research report. A blob. The script was submitting tasks with output_schema: { type: "auto" }. Auto mode returns structured JSON — right for batch enrichment pipelines. The script had a second stage that called Bedrock to reformat the JSON into markdown. When that Bedrock call failed silently, the fallback wrote the raw JSON directly to the file. No error surfaced. The fix: Parallel's Task API has a "text" mode. It returns a full markdown narrative report with inline citations, no schema definition required, no downstream formatting step. The change: "auto" to "text". One word. The Bedrock formatting step — 36 lines of bash, temp files, jq templates, an aws bedrock-runtime invoke-model invocation, fallback logic, cleanup — was deleted. Fewer parts. One fewer API call per research run. No silent fallback.

Three Bugs

BugProblemFix
IAM mismatch Script used openclaw-wiki-* job names; IAM allowed openclaw-voice-* Align script to working prefix
Non-ASCII in Bedrock Em-dashes in prompt template rejected by AWS CLI --body file:// fileb:// — one character
GHA metadata Grep pattern matched body content, not frontmatter; multi-line JSON broke step output Python frontmatter parser + heredoc delimiters

After those three fixes: pipeline ran end to end.

With ingest-voice.sh deployed and AGENTS.md routing voice memos through the full pipeline, a test memo went in. It failed immediately. Bug 1: ingest-voice.sh was using openclaw-wiki-* job names for AWS Transcribe. The IAM policy on the instance role allowed openclaw-voice-*. These two had never been tested together. Bug 2: wiki-integrate.sh built a JSON request body that contained em-dashes and Unicode. The AWS CLI's --body file:// flag treats the file as text and rejects non-ASCII characters. Fix: fileb:// instead of file://. One character. Bug 3: The GitHub Actions workflow used grep to extract frontmatter fields. The pattern matched body content, not frontmatter. And the summary extraction piped multi-line JSON into echo which GitHub Actions can't parse as step output. Fix: a Python-based frontmatter parser scoped to the YAML block, and heredoc delimiters for all GITHUB_OUTPUT writes. After those three fixes, the pipeline ran end to end.

The laptop was deploying old code.

Three Days

Two variants of the same cycle. Bot-driven when it could fix things itself. Human-driven when the environment got too noisy.

All of this took roughly three days. The actual process was not elegant in a traditional software-engineering sense. It was a loop: catch an error, feed it back into the system, watch the fix attempt, verify or redirect. Sometimes the bot would fix the issue itself and I'd tell it to push upstream. Other times I'd copy Telegram outputs — error messages, stack traces, the bot's own confused responses — back into Cursor, let the model reason through the issue, and apply the fix from there. That loop was the real engine of progress. There were moments when the environment fought back. The SSM-based interaction would sometimes overwhelm Cursor's terminal session memory. When that happened, I told the AI to stop executing and give me the exact commands to run in sequence. I'd run them myself, gather the debugging output, and feed the results back. This is not a failure mode worth hiding. It's a realistic human-agent handoff pattern.

Two Loops

The phone loop runs daily. The laptop loop runs weekly. Both write to the same repos.

The three-day build settled into something steadier. The system now runs on two rhythms. The phone loop runs daily: Ford sends voice memos, questions, research commands. The bot transcribes, publishes, replies. Every interaction adds to the knowledge graph. The graph grows without anyone managing it. The laptop loop runs weekly: pull the latest from both repos — including any patches the bot committed directly. Review what changed. Tune the workspace files. Edit the tool scripts. Run the 196 Bats tests. Ship. Both loops write to rvaopenclaw and openclaw-research. The bot is on the control plane too — it commits fixes, pushes patches, keeps the KB current. The human reviews. Corrections go back through git. The loop closes.

What It Is Now

One EC2 instance. One gateway process. All access via SSM. Bedrock called on every message — no inference on the instance itself.

One EC2 instance. One Node.js gateway. Port 18789, localhost only. All external access through AWS Systems Manager. On every message: six markdown files assembled into a prompt, session history appended from a JSONL file on local EBS, Bedrock called via the instance's IAM role, response returned. The model is not on the instance — the instance has 4GB of RAM and runs no inference at all. It's a prompt assembler and a caller.

The AWS bill shows $0 right now — free credits absorbing the real spend. The actual cost is about $125 a month at the usage levels in this series: 391 Bedrock invocations in seven days, short voice clips hitting the 15-second Transcribe billing minimum, one to three research tasks a day. That's lower than the heavy-use estimate. The gap is prompt caching. Bedrock caches the system prompt — the 6,650 tokens of workspace files — so most input arrives at $0.30 per million tokens rather than $3.00. Cache writes cost more but happen once per session. The net result is that observed Bedrock cost came in at about $28 for the week rather than the $44 the raw-input estimate would predict. The system is cheaper in practice than on paper. At light use: about $40 a month all in. The model choice is still the largest lever. Nova 2 Lite would cut the Bedrock line by 8.6×. But as Part 2 documented, Nova 2 Lite can't follow the workspace instructions. That saving would cost you the assistant.
