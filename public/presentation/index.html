Then WhatsApp Closed the Door.

How a health monitor turned 20 gateway restarts into a rate-limited phone number.

The health monitor turned every restart into a WhatsApp reconnect, a 401, exponential retries. Twenty restarts. Hundreds of failed auth attempts. Phone: "Can't link new devices at this time."

Act 2

Teaching It to Think

The bot was running. Every session started fresh. No memory. And something worse — it was ignoring instructions I'd written for it.

The Programming Language is English.

SOUL.md — personality
USER.md — who you are
TOOLS.md — available tools
AGENTS.md — operating manual
MEMORY.md — long-term facts

Injected into every prompt.
Change a file → next message follows it.
No restart. No redeployment.
No config change.

The runtime is a language model. The deploy target is a filesystem.

This is the core insight. Six markdown files on disk, concatenated into the system prompt on every message. To change the personality — edit SOUL.md, run make ship, send a message. The next response follows the new instructions. You're writing software. It just happens to be in English.

SOUL.md

ok but this is actually easy. do this: ...

you're kinda overcomplicating it. just: ...

wait... no. do it like this: ...

Sharp. Lowercase. Slightly unimpressed. Efficient.
Not a technical innovation — an ergonomic one.
For the amount of time you'll spend talking to this thing, tone matters.

This is the actual personality spec. The model talks like a Gen Z teenager who finds your question mildly beneath them but answers correctly anyway. Not a profound technical achievement. But the bot felt like someone with opinions, not a customer service form.

Memory Has Layers.

Raw sources stay locked. The wiki is the working layer. Agent Governance — the workspace files — runs on every turn.

The temptation is to load everything into context. At 500 research files, that's over a million tokens — the entire context window, nothing left for conversation. The spec named this failure mode the "Token Crusher." Instead: a flat JSON index, ~15k tokens, fetched on demand. Load the card catalog, not the library. The model finds the entry, fetches the document, reads it.

The Idea.

A voice memo becomes a wiki node. Nothing about the implementation was this clean.

The concept is simple. The execution had seven steps, each one that could fail independently. Let me show you the pipeline.

Nine Steps.

Transcription error in step 3. Didn't surface until step 6.

Upload to S3. Start a Transcribe job. Poll until done. Download transcript. Create source page. Run LLM integration pass. Commit everything. Index rebuilds. Notify. Nine steps. The error lives in step 3 — but you don't find out until step 6 when the wiki page says your son was born in Lagos, Nigeria. More on that in Act 3.

The Pipeline, End to End.

Telegram → S3 → Transcribe → markdown → GitHub → knowledge base. $0.024/min. 60–120 seconds. Bot acknowledges in 2.

The acknowledgment is non-optional. "Got it, transcribing..." costs nothing and prevents two minutes of silence reading as failure. The pipeline runs in the background. The bot handles other messages while it waits.

The Keys That Weren't There.

Three paths for API keys. Only one reached the running gateway process.

First research run. Script called the API. Nothing came back. The .env file: PARALLEL_API_KEY= — empty string. The key had been in SSM Parameter Store since day one. Nothing had ever pulled it into the environment. Then the real gap: .env was correct, .profile source line was there, interactive sessions picked up the keys fine — but the gateway runs as a systemd user service. Systemd doesn't source .profile. One line in the service file: EnvironmentFile=/home/ubuntu/.openclaw/.env. Verified from /proc//environ. This is in the takeaways.

Same Instructions. Different Model Class.

Nova 2 Lite ignored the workspace files. Sonnet 4.6 followed them. No prompt engineering closes that gap.

The log: "agent model: amazon-bedrock/global.amazon.nova-2-lite-v1:0." Nova 2 Lite. Not Sonnet 4.6. The model upgrade had never been saved into the running config. Every session since day one had been the wrong model. Nova 2 Lite is optimized for speed — AWS's own docs say "system prompt adherence can decline slightly as context increases." With 6,650 tokens of workspace files on every turn, "slightly" is not the word. This is not a tier difference. It's a model-class difference. No amount of prompting closes it.

sed -i "s|nova-2-lite-v1:0|us.anthropic.claude-sonnet-4-6|" openclaw.json

[gateway] ready (9 plugins, 1.9s)

check what model is running before touching the instructions.

One string replacement. Gateway restart. The personality took hold. The protocols started working. The capabilities unlocked. The lesson: making instructions louder doesn't help. Checking the logs does.

Act 3

What Broke

The right model was running. The pipeline was deployed. The first real test produced wrong data, and the bot was about to fix its own code without telling anyone.

Correct Reasoning. Bad Data.

wiki/entity/rio.md
Birthplace: Lagos, Nigeria

speculative: was Ford living in Nigeria at the time of the birth?

AWS Transcribe heard "Lagos" where Richmond was said.
The LLM had no way to know. It reasoned correctly from bad data.
"Thadeus" became "Thaddeus" everywhere. Profile created as will-prior.md.

The pipeline worked. Within three minutes, five wiki pages were created, committed to GitHub, bot replied with a link. One page said my son was born in Lagos, Nigeria. Reasonable inference from bad transcription. This isn't a bug to fix with prompting. Speech-to-text has error rates. LLM passes amplify them. The mitigation is a workflow: bot creates drafts, human reviews, corrections commit with provenance. Source transcripts stay immutable. Derived pages evolve.

One Parameter. Deleted an Entire Stage.

"auto" → "text" deleted the Bedrock reformatting stage and its silent failure mode.

First deep research results rendered as 250 lines of raw JSON on GitHub. The script was submitting with output_schema: { type: "auto" } — structured JSON, right for batch pipelines, wrong for readable reports. There was a second stage that called Bedrock to reformat it. When that failed silently, the fallback wrote raw JSON directly to the file. The fix: Parallel's API has a "text" mode. Returns markdown with inline citations, no schema needed, no downstream formatting step. One word change. 36 lines of bash deleted.

The Bot Fixed Its Own Code.

Debugging its own git retry logic — then immediately ingesting a voice memo about Dune and The Matrix.

lib/git.sh had no retry loop. Push rejections from concurrent GitHub Actions index rebuilds were failing with status: partial. The bot diagnosed the issue, wrote a retry-with-rebase loop, and committed it. On EC2. It did not push. The fix sat on the instance until a git pull surfaced commit fdcef68. Correct diagnosis. Clean fix. Nobody knew about it. This is what happens when an agent has write access to its own infrastructure but no deployment discipline.

Research Arrives in Telegram.

Index rebuilt. Notification delivered. The loop closes in under ten minutes.

A research task runs, results push to the GitHub repo, Actions rebuilds the index, SSM sends a notification to the bot, bot delivers it to you. Under ten minutes from trigger to delivery. The system confirms itself — no polling, no manual refresh. It tells you.

Skills: The AI Doesn't Start From Scratch.

Same tokens, deeper thinking. Without skills, the AI explores broadly and fails often. With skills, the early branches are pre-solved.

Every time I figured out how to do something hard, I asked the AI to write a skill file capturing the process. Seven total by the end: deploy, operate, WhatsApp recovery, customize, personality, research, deploy-changes. These don't run on the bot — they're for the human-AI pair working in Cursor. Next week, the AI picks up where the last session left off. No rediscovery. The system compounds instead of resetting.

The Bot Needs an Onboarding Doc.

CLAUDE.md — read before every Cursor session

  • Always push code changes after committing
  • Always pull the KB before reading it
  • Never leave uncommitted changes silently
ship:
    git push origin main && make deploy

The gap between "on GitHub" and "on EC2" was bridged by human memory.
make ship closes it.

The bot is capable of fixing its own infrastructure. It cannot be trusted to follow deployment discipline without explicit written rules. CLAUDE.md is the onboarding doc. make ship is the atomic operation. One command: push to GitHub and sync to EC2. The gap no longer exists.

Three Days.

Two variants of the same cycle. Bot-driven when it could fix things itself. Human-driven when the environment got too noisy.

All of this took three days. Catch an error, feed it back into the system, watch the fix attempt, verify or redirect. Sometimes the bot fixed the issue itself. Other times I'd copy Telegram outputs back into Cursor and let the model reason through it. When the environment got overwhelming, I told the AI to stop executing and give me the exact commands to run in sequence. That handoff pattern is not a failure mode. It's how you keep moving.

Two Loops.

The phone loop runs daily. The laptop loop runs weekly. Both write to the same repos.

The build settled into two rhythms. Phone loop: daily use, voice memos, questions, research commands. The knowledge graph grows without anyone managing it. Laptop loop: weekly, pull the latest — including patches the bot committed directly — review, tune workspace files, run 196 Bats tests, ship. Both loops write to both repos. The bot is a developer on its own codebase.

What It Is Now.

One EC2 instance. One gateway process. All access via SSM. Bedrock called on every message — no inference on the instance itself.

One EC2 instance. One Node.js gateway. Port 18789, localhost only. All external access through SSM. On every message: six markdown files assembled into a prompt, session history appended from JSONL on EBS, Bedrock called via IAM role, response returned. The instance has 4GB of RAM and runs no inference. It's a prompt assembler and a caller. The model is in Bedrock.

The Librarian Protocol.

Fetch index.json from GitHub raw URL. Filter by title, tags, summary. Return top matches with links.

When you ask about past research, the bot follows the Librarian Protocol in AGENTS.md: fetch the index — about 15k tokens for the full corpus — filter by query, tags, and summary, return the top matches. No vector database. No embeddings. Just a model reasoning over a structured list. If nothing matches, it proposes a specific follow-up query based on what's missing.

Deep research is triggered by a message. The bot acknowledges immediately, fires a Parallel AI task, and sends the result when ready — usually within a few minutes. Markdown with inline citations. Published to the research repo. Linked back in the conversation. The bot doesn't just summarize. It publishes. Future queries can find it.

Six Things to Steal.

  1. Work with the platform's design, not against it. CLI pairing works. WebSocket tunnels don't.
  2. Check what model is actually running before touching the instructions.
  3. Systemd doesn't source .profile. If your gateway has no env vars, check the unit file.
  4. Simpler pipelines fail louder. "auto" to "text" deleted a stage and its silent failure mode.
  5. Your agent needs an onboarding doc. Write it in plain English. It will follow it.
  6. ship = push + deploy. Close the gap or you'll forget it exists.
Each one is a real failure that happened and got fixed. Not theoretical. The goal is that you leave with at least one of these applying to something you're building.
